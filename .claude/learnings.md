# Pilcrow v1 — Agent Learnings

Append-only. Format: `- [YYYY-MM-DD] [agent or area]: [the lesson, one sentence]`

- [2026-04-29] typography-architect: Pretext's `continueSoftHyphenBreakableSegment` packs graphemes after a soft-hyphen, so the materialized line can end with `-<suffix>` (e.g. `ital-i`) not just `-`; orphan detection must handle both cases, not only lines that end with a bare `-`.
- [2026-04-29] typography-architect: Stripping SHYs in document order (first SHY first) causes collateral grapheme-break regressions on unrelated words; always use targeted stem-search (`stem + U+00AD`) to find the specific SHY to strip.
- [2026-04-29] typography-architect: The orphan guard helpers (`guardFlat`, `guardRich`) must be defined inside `page.evaluate()`'s callback — they run in the browser context and cannot be imported as Node modules; keep them co-located in `playwright.ts`.
- [2026-04-29] typography-architect: When adding `<cite>` (pull quote attribution) to the HTML, Hyphenopoly will hyphenate author names inside `<cite>` unless `<cite>` is explicitly added to the skip list in `hyphenate.ts` alongside `<code>` — author names must never be hyphenated.
- [2026-04-30] design-critic: Screenshot pattern-matching for hyphenation orphans is unreliable — a hyphenated suffix at line-end (e.g. `prim-`, `destina-`, `genu-`, `walkRichIn-`) reads visually as a standalone fragment but is actually the tail of a 100%+ fill line; char-count `.pt-line` widths is the authoritative method (two incidents: `walkRichIn-` on `inline-markup.md` ¶3, and five false-positive orphans across all posts — all resolved as full-fill lines under char-count inspection); design-critic.md Critical rule 7 now mandates char-count first.
- [2026-04-29] typography-architect: The `splicePostBody` regex in `pilcrow-typeset.ts` used a non-greedy `[\s\S]*?` match that stops at the FIRST `</div>` inside `.post-body`; any primitive that introduces a nested `<div>` inside the post body (e.g. the `.footnotes-mark` pilcrow glyph element) will break it — replace with a depth-counting balanced-div scanner, not a regex.
- [2026-04-29] typography-architect: CSS `::before { content: 'U+glyph' }` with Unicode characters (¶, *, etc.) is read aloud by NVDA ("pilcrow"), JAWS, and VoiceOver when verbose; always use a real DOM element with `aria-hidden="true"` for decorative Unicode glyphs instead of generated content.
- [2026-04-29] typography-architect: Astro includes `remark-gfm@4.0.1` by default with `gfm: true` — GFM footnotes work without adding it to `astro.config.mjs`; do not add it as an explicit plugin as it will run twice.
- [2026-04-30] typography-architect: CSS Grid grid-column can only target direct children of the grid container — an `<aside>` nested inside `<span>` inside `<p>` cannot use grid-column directly; use float + negative margin-right (`calc(-width - gap)`) instead, which is the classical magazine-float technique and works with no JS.
- [2026-04-30] typography-architect: When `.post-body` gains a Grid wider than the `.post` wrapper's `max-width: 65ch`, the right-margin column overflows or is clipped; the fix is to expand `.post` to `max-width: calc(65ch + gap + sidenote-width)` at the relevant breakpoint in a `@media (min-width: breakpoint)` rule — not by removing the prose max-width.
- [2026-04-30] typography-architect: The `readMeasurementCSS()` fallback for `postBodyMaxWidth` is `'65ch'` — if `.post-body` loses its `max-width` property (e.g. after Grid restructuring), the fallback still correctly constrains body paragraphs in the loader page to 65ch; no change to the extraction logic is needed as long as the fallback matches the prose column width.
- [2026-04-30] typography-architect: Astro 6 content collections (glob loader) use a persistent data store cache at `node_modules/.astro/data-store.json` (devalue-serialized) — when the digest of a source file hasn't changed, Astro returns the cached rendered HTML and NEVER calls the markdown processor; a rehype plugin that appears not to execute is almost always a cache hit, not a pipeline bug; clear `node_modules/.astro/data-store.json` (not just `.astro/` or `node_modules/.vite`) to force a fresh render.
- [2026-04-30] typography-architect: Errors inside rehype plugins running in Astro's content-layer glob loader are caught silently by the loader's try/catch and logged only as `[glob-loader] Error rendering X.md: ...`; the build succeeds but `entry.rendered` is set to `undefined`, producing empty post bodies — always run a full clean build with `node_modules/.astro/data-store.json` deleted when debugging plugin execution.
- [2026-05-01] typography-architect: npm strips files named `.gitignore` when packing the package — when bundling a template directory inside an npm package, store the template's gitignore as `gitignore` (no leading dot) and have the CLI rename it to `.gitignore` when copying into the user's target directory; affects any `create-*` npm package that needs to ship a `.gitignore` in its template.
- [2026-05-01] typography-architect: `packages/` directory inside an Astro project is automatically ignored by Astro's build pipeline (Astro only processes `src/`, `public/`, and configured directories) — no monorepo workspace config needed when keeping a sibling create-package alongside the main Astro project.
- [2026-05-01] typesetter: Sidenote markers (`<sup class="sidenote-marker">`) appended AFTER `display:block` `.pt-line` spans inside `<p>` trigger CSS anonymous block wrapping, rendering the marker as a standalone element on its own visual line ("stray 1 in accent colour"); fix is to inject the marker HTML inside the last `.pt-line` span via `buildLineSpansHTML(inners, markers)` helper in `playwright.ts`.
- [2026-05-06] typesetter: Astro 6 + `moduleResolution: 'Bundler'` does NOT auto-pick up `@types/node` from a workspace-hoisted devDep alone — explicit `"types": ["node"]` in the package's `tsconfig.json` was required alongside the devDep declaration to resolve `node:fs/promises`, `process`, etc. (Discovered during `pilcrow-typeset` extraction.)
- [2026-05-06] typesetter: `unist-util-visit` v5 with hast trees passes `Root | Element | undefined` as the `parent` argument to the visitor callback, NOT `Parent | null`. The codebase had typed it as `Parent | null` since the original write — the typed package extraction surfaced the mismatch. Pattern for any future `visit()` callback against a hast tree: import `Root` and `Element` from `'hast'`, type parent as `Root | Element | undefined`, and guard with `parent === undefined`.
- [2026-05-06] typesetter: Sprint extraction sub-tasks can land their work eagerly during file moves rather than as discrete steps — always read the current state of the target file before planning a write, the spec may already be satisfied. Cosmetic churn against an API-locked surface is worse than no edit. (Verified during `pilcrow-typeset@0.1.0` sub-task 3: `typeset()` and four re-exports were already in place from sub-tasks 1–2; correct response was to verify line-by-line and confirm gates, not rewrite.)
- [2026-05-06] typesetter: When verifying that a build-time text transformation actually ran (Hyphenopoly soft-hyphen injection, or any pass that emits markers a downstream consumer may eat), grep the entire `dist/` corpus, not a single representative file. A single-post grep can return zero matches even when the transformation worked correctly, because every emitted marker can land at a break position where the downstream pass (in this case pretext) consumes it back out. Corpus-wide grep is the only reliable signal. (Discovered during sub-task 4 verification: `dist/posts/the-cheapest-signal/index.html` had 0 U+00AD characters and almost triggered a false-positive "stop and report"; all-posts grep surfaced 20 surviving SHYs across the corpus, plus mid-syllable line breaks in every post — the actual proof.)
- [2026-05-06] typesetter: Sprint-shaped work (multi-deliverable, time-boxed, sub-tasks under ~30 minutes each) does not need per-sub-task feature specs. The sprint plan itself functions as the spec. Reserve `context/feature-specs/NN-<name>.md` for unsprinted standalone features where the planning artefact and the spec artefact would otherwise be the same document written twice.
- [2026-05-06] typesetter: Cross-package sibling-relative imports (e.g. `'../../pilcrow-typeset/dist/index.js'`) work for monorepo dev where the consuming package's dist/ is co-located with the dependency's dist/, but break the moment the consuming package is consumed via `npm install` — the relative path is preserved verbatim in the published tarball and resolves to nothing in the consumer's `node_modules` layout. For published sibling packages within the same monorepo: declare the dep explicitly via the package name in `dependencies`, and use `import { X } from '<package-name>'`. For local dev pre-publish, declare the dep as `"file:../<package>"` in `dependencies` (single declaration only — see next learning). Mechanically swap `file:..` → `^<version>` via `npm pkg set` immediately before `npm publish`, then revert after — Stream D's "swap-publish-revert" sequence in PILCROW_V3_SPRINT_PLAN.md captures the pattern.
- [2026-05-06] typesetter: Bun's resolver does NOT treat a `devDependencies` `"file:..."` entry as an override of a duplicate `dependencies` `"^x.y.z"` entry. It warns about the duplicate declaration, then attempts registry resolution of the `dependencies` entry first, failing with a 404 if the package isn't on npm yet. This may differ from npm/pnpm/yarn behaviour — do not assume. Use a single declaration in `dependencies`; switch the version specifier between `file:..` (local dev) and `^x.y.z` (publish) at publish time rather than dual-declaring.
- [2026-05-06] typesetter: Astro 6 emits inline `<script type="module">` blocks inside `.astro` files verbatim — bare specifiers like `import * as pretext from '@chenglou/pretext'` will 404 in the browser because no import map is generated. Pretext (and any other client-bundled module) must live in a separate `.ts` file imported via a non-`is:inline` `<script>` tag from the .astro file; that's the only way Vite resolves the bare specifier and emits a real client chunk.
- [2026-05-06] typesetter: Astro 6 treats files matching `src/pages/_*.astro` as private and does not build them — using an underscore-prefixed filename for a "hidden" spike route makes a real-browser load impossible. Use a non-prefixed filename + `<meta name="robots" content="noindex,nofollow">` + an explicit sitemap-filter exclusion in `astro.config.mjs` instead.
- [2026-05-06] typesetter: For a temporary route to be excluded from the sitemap, add a `filter:` clause to `@astrojs/sitemap` config that returns `false` when the URL contains the route slug — and document the deletion path inline (e.g. "delete this clause when the spike file is removed") so the workaround does not outlive its purpose.
- [2026-05-06] typesetter: Pretext relies on `Intl.Segmenter` for grapheme/word segmentation; Firefox 125+ (April 2024) is the gate. Detect with `typeof Intl?.Segmenter === 'function'` and ship a fallback notice rather than a silent failure when running pretext in client code.
- [2026-05-06] typesetter: `@chenglou/pretext@0.0.6` ships at ~16 kB gzipped to the client bundle and has zero transitive runtime deps — it's cheaper than most analytics scripts and there is no architectural reason to gate it behind a hosted backend.
- [2026-05-06] typesetter: `await document.fonts.ready` is mandatory before any pretext call in the browser — same gate `playwright.ts` uses. Without it, canvas `measureText` returns metrics for the fallback system font and pretext breaks lines for the wrong typeface; the symptom is a visually correct page that diverges from the deployed pretext output by 1–3 px per line.
- [2026-05-06] typesetter: To derive the canvas font shorthand and lineHeight in the browser exactly the way `playwright.ts` does, attach a probe `<p>` inside `.post-body`, read `getComputedStyle()`, then build `${cs.fontStyle} ${cs.fontWeight} ${cs.fontSize} ${quotedFirstFamily}`. CSS `line-height: 1.7` resolves to a px value via `parseFloat(cs.lineHeight)` — no manual multiplication needed in modern browsers.
- [2026-05-06] typesetter: Browser canvas + Playwright canvas use the same `OffscreenCanvas.measureText` + `Intl.Segmenter` path, so output is byte-identical when inputs match. The three structural differences between a naive in-browser pretext call and the deployed pipeline are: column width (must come from `--prose-measure` in `global.css`, not the visual element width), drop-cap float-aware narrowing (must call `layoutNextLineRange` per line and shrink width while `y < capHeight`), and SHY pre-pass (Hyphenopoly's soft-hyphen injection must run before pretext sees the text).
- [2026-05-06] typesetter: The installed `hyphenopoly@6.1.0` package's main entry (`hyphenopoly.module.js`) uses `WebAssembly` + `TextDecoder` and a fully configurable async `loader(file, baseURL)` — no Node-specific imports — so it CAN run in a browser if (a) the package is moved from devDependencies to dependencies (so Vite bundles it), (b) `en-gb.wasm` is shipped at a public URL like `/_hyphenopoly/en-gb.wasm`, and (c) the loader is rewritten to `fetch().then(r => r.arrayBuffer())`. The companion `Hyphenopoly_Loader.js` is NOT a programmatic API — it's a `hyphens: auto` CSS polyfill that mutates the DOM by class selector and cannot be invoked synchronously over a string. So any browser SHY pre-pass must use `hyphenopoly.module.js` with a custom fetch loader, not `Hyphenopoly_Loader.js`.
- [2026-05-06] typesetter: `hyphenopoly@6.1.0` ships only `<lang>.wasm` files (e.g. `en-gb.wasm` at 33 kB raw / 22 kB gzipped) — there are NO separate `.hpb` pattern files. Pattern data is embedded in the wasm. The loader callback only ever fetches `<lang>.wasm`. Don't waste time looking for `.hpb` to ship.
- [2026-05-06] typesetter: First-typeset Hyphenopoly init in browser (lazy-loaded wasm fetch + wasm instantiation + en-gb language registration) is ~6–20 ms in headless Chromium. Cheap enough to gate behind the first `typeset()` call rather than eager-loading at import time — the latency disappears against the typesetter's own ~30 ms work and never blocks initial paint.
- [2026-05-06] typesetter: BrowserRenderer step 3(a) (`resolveProseMeasurePx()` reading `--prose-measure`) is necessary as a hard-error invariant check, but NOT sufficient as the per-paragraph layout width — `playwright.ts:930` uses `p.clientWidth` per paragraph, which is the correct semantics for narrowed containers (`aside.sidenote 25ch`, `.pullquote 50ch`, `.footnotes p`). Using a single `--prose-measure`-derived value across all paragraphs causes BrowserRenderer to lay out sidenote and pull-quote paragraphs at full prose width, breaking parity with the production pipeline. Mirror the production semantics: `resolvedWidth = options.maxWidth || p.clientWidth || proseMeasurePx` per paragraph.
- [2026-05-06] typesetter: BrowserRenderer acceptance gate against deployed `pilcrow.page` is two-axis: (a) BrowserRenderer ≡ local-built dist HTML (proves byte-identical to current production pipeline), (b) BrowserRenderer ≡ deployed HTML (proves the deployed site is current). When (a) passes but (b) fails, the divergence is a stale deploy, not a renderer bug. Always run BOTH comparisons and report which axis diverged. If only (b) fails, redeploy is the fix; BrowserRenderer is correct.
- [2026-05-06] typesetter: Hyphenopoly v6 embeds language pattern data inside the `<lang>.wasm` binary; there is NO companion `.hpb` file shipped with the package. Earlier sprint plans expected to ship `en-gb.hpb` alongside `en-gb.wasm` — that expectation is wrong, only the wasm needs to land in `public/_hyphenopoly/`. Verified `find node_modules/hyphenopoly -name '*.hpb'` returns empty; only `<lang>.wasm` files exist.
- [2026-05-06] typesetter: `resolveProseMeasurePx()` returning a single number for all paragraphs is structurally wrong — production `playwright.ts:930` reads `p.clientWidth` per paragraph, which gives sidenote-narrowed (25ch), pullquote-narrowed (50ch), footnotes-narrowed widths their correct layout target. BrowserRenderer must mirror that exactly: `resolvedWidth = options.maxWidth || p.clientWidth || proseMeasurePx` (the `--prose-measure` invariant check is retained as fallback / sanity rail, not as the source of truth).
- [2026-05-06] typesetter: The 2026-04-29 acceptance-gate rule "compare against deployed" is symmetric — it catches drift in either direction. The inverted-asymmetry case (current local is correct, deployed is stale) reads superficially like a gate failure, but the resolution is to redeploy and re-run, not to relax the rule. The rule itself is sound; the asymmetry is in the artefact under test, not the methodology.
- [2026-05-06] typesetter: BrowserRenderer gate uncovered Linux Playwright Chromium 147 silently failing to load variable-axis TTF fonts in headless contexts despite bytes being delivered with correct MIME and CORS. Symptom: rendered glyph widths returned as clean integers (10, 170, 264) on Linux vs sub-pixel-precise floats (9.8378, 168.077, 263.713) on macOS arm64. FONTS-API-PROBE confirmed FontFace.status: 'loaded' on Linux despite no actual rendering. Diagnosis required parallel-family probe with both variable and static instances under throwaway @font-face declarations. Fix: switched body type pipeline from variable Fraunces to four static 144pt opsz instances (Regular 400 / SemiBold 600 / Bold 700 / Italic 400 italic). Drop-cap weight changed 500 → 600 because static release has no weight 500 (variable surface synthesised it from the wght axis); SemiBold preserves drop-cap-vs-body weight distinction. Residual ~3% line-count drift on long posts (>100 pt-lines) remains due to genuine CoreText-vs-FreeType sub-pixel rasterisation differences — documented as a known limitation, candidate for v1.x Linux-build-container approach. Asset surface shrank 803 KB variable → 389 KB static. The 2026-04-29 deploy-comparison rule's restatement: BrowserRenderer must match local Playwright dist/ byte-for-byte; deployed pilcrow.page may drift up to +3 lines on long posts. CF Pages build cache also surfaced as a stale-output source during this arc — clearing via dashboard + empty-commit-redeploy resolves. Wrangler is the right tool for verifying CF Pages deployment status; installed during this work.
- [2026-05-06] typesetter: Diagnostic harnesses must inherit from production CSS, not duplicate it. The spike-browser-renderer-gate page hard-coded its own Google Fonts <link> during the original spike, which became invisible drift when the production fix swapped fonts to self-hosted statics — the gate then measured BrowserRenderer against the OLD variable-Fraunces baseline (114 lines) instead of the new static baseline (102 lines), producing a false-fail of 114-of-114. Pattern to remember: any harness that compares X against canonical-Y must reach canonical-Y by reference, never by reproduction. If you find yourself copy-pasting font links, build commands, or measurement-critical CSS into a diagnostic, that copy is a future false-positive waiting to happen. The fix here was deleting six lines from one page; the lesson is wider — never reproduce production state inside diagnostic infrastructure.
- [2026-05-06] typesetter: document.fonts.ready resolves based on which @font-face declarations have been REQUESTED (i.e. needed by laid-out content), not which have been DECLARED. If a weight/style is only used by content that's about to be programmatically injected (e.g. a drop-cap span styled with font-weight: 600 when no other element on the page uses 600), document.fonts.ready can resolve while that weight is still unrequested. Subsequent getBoundingClientRect on the injected element falls back to the closest loaded weight rendered with synthetic transformations — measurably different geometry than the real cut. BrowserRenderer hit this for SemiBold 600 cap measurement (78.64px synthetic-bold rect vs 71.14px correct rect). Fix: explicit Promise.all([document.fonts.load('600 87.4px Fraunces'), document.fonts.load('400 19px Fraunces'), ...]) for every declared weight/style after document.fonts.ready, before any injection-and-measure path. playwright.ts didn't hit this naturally because its per-paragraph processing interleaved enough time for lazy font fetches; a single-shot measurement path won't. Diagnostic tell: the cap rendered as a square (78.64×78.64) instead of an advance-width-driven rectangle (71.14×78.64) — synthetic-bold widening tends to push glyphs toward the line-height square because the synthesis is uniform-axis.
- [2026-05-06] typesetter: When a structural side-by-side comparison surfaces multiple deltas (e.g. position:absolute vs static, visibility:hidden vs visible, body padding, viewport size), the most-different-looking delta isn't necessarily the causal one. The cap-measurement debug round identified four genuine structural differences between BrowserRenderer's host setup and playwright.ts's loaderHTML; the actual cause was below all of them, in the font-loading lifecycle. The corrective signal came from the extended probe — both renderers reporting display: block + float: left identically despite different rects — which forced 'cause is below the cap's own layout rules' as the conclusion and redirected to font-loading. Lesson: structural reads are necessary but not sufficient. When a probe matrix shows 'identical CSS, different rect', look at the lifecycle (when was the element created, when was measurement called, what async work might still be pending) before the structure. Defaults to suspect: font loading, image decoding, layout-shift settling, microtask flush ordering.
